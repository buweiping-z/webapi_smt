## 一、数据库变更

### 1.1 新表：`qualified_inspectors`

点检资格人员表，存储具备点检资格的人员信息。

| 列名 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `id` | INT | PK, AUTO_INCREMENT | 自增主键 |
| `employee_id` | VARCHAR(9) | NOT NULL, UNIQUE INDEX | 工号，格式"SN"+6位数字，如"SN000001" |
| `last_name` | VARCHAR(6) | NOT NULL | 姓，最长2个中文字（UTF-8下每中文字3字节，6字节足够） |

**建表SQL（MySQL）**:
```sql
CREATE TABLE qualified_inspectors (
    id INT AUTO_INCREMENT PRIMARY KEY,
    employee_id VARCHAR(9) NOT NULL,
    last_name VARCHAR(6) NOT NULL,
    UNIQUE INDEX idx_employee_id (employee_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

**Model 类** (`Models/QualifiedInspector.cs`):
```csharp
using System.ComponentModel.DataAnnotations.Schema;

namespace webapi.Models
{
    [Table("qualified_inspectors")]
    public class QualifiedInspector
    {
        [Column("id")]
        public int Id { get; set; }

        [Column("employee_id")]
        public string EmployeeId { get; set; } = string.Empty;

        [Column("last_name")]
        public string LastName { get; set; } = string.Empty;
    }
}
```

### 1.2 修改表：`inspection_templates`

新增一列：

| 列名 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `is_mandatory` | TINYINT(1) | 1 | 1=必须点检，0=选择点检 |

**迁移SQL**:
```sql
ALTER TABLE inspection_templates
ADD COLUMN is_mandatory TINYINT(1) NOT NULL DEFAULT 1;
```

现有数据全部默认为必须点检，向后兼容。实施后需人工审查模板数据，将确实为"选择点检"的项目手动更新为 0。

**Model 属性**（追加到 `Models/InspectionTemplate.cs`）:
```csharp
[Column("is_mandatory")]
public bool IsMandatory { get; set; } = true;
```

### 1.3 实体关系

```
qualified_inspectors.employee_id  ←→  inspection_records.employee_id
inspection_templates.is_mandatory →  漏检统计逻辑
```

## 二、API 端点

### 新增端点总览

| 方法 | 路由 | 用途 | 需求 |
|------|------|------|------|
| GET | `api/inspection/operators/list` | 获取点检资格人员列表 | #2 |
| POST | `api/inspection/operators/add` | 新增点检资格人员（支持单个/批量） | #2 |

### 2.1 `GET api/inspection/operators/list`

获取全部点检资格人员。

**响应**:
```json
[
  { "employeeId": "SN000001", "lastName": "张" },
  { "employeeId": "SN000002", "lastName": "李" }
]
```

### 2.2 `POST api/inspection/operators/add`

新增一个或多个点检资格人员。

**请求**:
```json
[
  { "employeeId": "SN000001", "lastName": "张" },
  { "employeeId": "SN000002", "lastName": "李" }
]
```

**响应（成功）**:
```json
{ "success": true, "message": "已添加 2 名点检资格人员" }
```

**响应（工号重复）**:
```json
{ "success": false, "message": "工号 SN000001 已存在" }
```

### 2.3 `DELETE api/inspection/operators/{employeeId}`

移除指定工号的点检资格。

**响应**:
```json
{ "success": true, "message": "已移除点检资格人员 SN000001" }
```

### 2.4 `GET api/inspection/operators/validate/{employeeId}`

验证工号是否具有点检资格。不做格式校验，仅查表。

**响应（有效）**:
```json
{ "valid": true, "employeeId": "SN000001", "lastName": "张" }
```

**响应（无效/不存在）**:
```json
{ "valid": false }
```

**安全措施**: 添加请求频率限制（API 层或中间件），防止暴力枚举工号。

### 2.5 `GET api/inspection/operators/daily?deviceModel=X&year=2026&month=6`

查询指定设备+月份内每日的点检者姓。

**逻辑**: 
1. 从 `inspection_records` 查找该设备当月记录（`inspection_time` 在当月范围内）
2. JOIN `qualified_inspectors` 通过 `employee_id`
3. 按日分组，取每天最新记录的 `employee_id` 对应的 `last_name`
4. 同一天有多次记录时，取最后一次的 `inspection_time`

**响应**:
```json
{
  "deviceModel": "设备A",
  "year": 2026,
  "month": 6,
  "inspectors": {
    "1": "张",
    "2": "李",
    "3": "张"
  }
}
```

`inspectors` 的 key 为日期（1-31），value 为点检者姓。无记录的日期不出现在响应中。

### 修改端点

### 2.6 `POST api/inspection/submit-full` — 增加工号验证

在保存记录前，验证 `request.EmployeeId` 是否在 `qualified_inspectors` 表中存在：

- 存在 → 正常保存
- 不存在 → 返回 `400 BadRequest`：`{ "success": false, "message": "该工号无点检资格" }`

不进行格式校验（如"SN"+数字），信任 `qualified_inspectors` 表的数据录入。

### 2.7 `POST api/inspection/submit`（旧端点）— 增加工号验证

同样在保存前验证 `request.EmployeeId`，保持与新端点一致。

### 2.8 `POST api/inspection/records/save` — 修复 + 改造

**预存Bug修复**: 创建新 `InspectionRecord` 时追加设置 `Frequency` 和 `PeriodKey`：

```csharp

var newRecord = new InspectionRecord
{
    EmployeeId = request.EmployeeId ?? "web",  // 改为使用请求中的工号
    DeviceModel = request.DeviceModel,
    DeviceName = request.DeviceModel,

```

前端保存时传入当前选定的点检者工号。

### 2.9 `GET api/inspection/frequency-summary` — 修改漏检统计逻辑

**原逻辑**: 检查设备是否有任何记录 → 有 = 已点检  
**新逻辑**: 对指定频率下所有 `is_mandatory=1` 的项目，逐一检查是否在周期内有对应的 `InspectionResult`

**统计口径**（明确）:
- 设备在该频率下所有必须点检项目（`is_mandatory=1`）全部有 InspectionResult → 该设备"已点检"
- 存在任一必须点检项目无 InspectionResult → 该设备"漏检"
- 选择点检项目（`is_mandatory=0`）不参与漏检判断
- 异常（×）点检结果仍然单独统计，不影响"已点检/漏检"分类

**统计步骤**（以"日"频率为例）:
1. 获取所有拥有"日"频率模板的设备列表
2. 对每台设备，查询其"日"频率下所有 `is_mandatory=1` 的模板项集合 M
3. 查询该设备当月的 InspectionRecord + InspectionResult
4. 若 M 中任一项目在当天无 InspectionResult → 该设备当日漏检
5. 汇总整月：设备在月内任意一天有漏检 → 计入 `uninspectedDevices`

**响应格式不变**:
```json
{
  "totalDevices": 5,
  "inspectedDevices": 3,
  "uninspectedDevices": 2,
  "uninspectedDeviceModels": ["DeviceA", "DeviceB"],
  "abnormalDevices": 1,
  "abnormalDeviceModels": ["DeviceC"]
}
```

## 四、错误处理

| 场景 | 处理 |
|------|------|
| 工号不在资格表中 | `submit-full` / `submit` / `records/save` 返回 400，前端提示"该工号无点检资格" |
| 设备无模板 | 返回空数组，前端显示"暂无点检项目" |
| 月份无点检记录 | `daily` API 返回空 inspectors，前端"点检者"行显示全空白 |
| 工号重复添加 | `operators/add` 返回 400，提示"工号已存在" |
| 删除不存在的工号 | `operators/{id}` 返回 404，提示"工号不存在" |
| `daily` API 设备不存在 | 返回空 inspectors，前端正常显示空行 |
| validate 请求频率过高 | 返回 429 Too Many Requests |

## 附录：预存 Bug 修复详情

**Bug**: `SaveDailyRecord` 创建新 `InspectionRecord` 时未设置 `Frequency` 和 `PeriodKey`

**位置**: `Controllers/InspectionController.cs` 约第229-237行

**影响**: 
- 同日第二次调用 `records/save` 时，因 `PeriodKey=""` 无法匹配已有记录（已有记录的 `PeriodKey` 也是空），创建重复记录
- 数据库唯一索引 `idx_records_period` 因两条记录 `PeriodKey` 均为空而失效（MySQL 对 NULL/空字符串的唯一索引行为）

**修复**: 在创建新记录时显式设置 `Frequency = frequency`（"日"）和 `PeriodKey = periodKey`（"yyyy-MM-dd"格式）
